5）是不是应该先按任务分类构造，再回头检查视频种类和原数据？

对，必须这么做。顺序不能反。

如果你先扫原视频再想任务，最后一定会被原数据分布绑架，得到一个“会看视频，但不会做 agent 决策”的模型。

我建议固定成 5 步：

先定义任务本体
给每类任务定目标比例
再盘点哪些视频类型能承载这些任务
决定每个桶是“派生”还是“补造”
最后才批量出样本
我建议的任务三维配额

单轮单答：50%
一问多答：20%
多问多答：20%
主动触发 / 持续监控：10%
记忆需求

按 episode 计：

当前可见即可答：55%
需要 think 但不需要 recall：25%
需要 think + recall：20%

推理类型

按任务计：

短时序 / 顺序：15%
程序步骤跟踪：15%

然后再映射视频类型

1、但有一个关键点要先改：固定视觉 buffer + 固定 think buffer 只能解决“工作记忆”，不能单独覆盖无限长视频。 你还必须加第三层：
Recent visual buffer：近端视觉工作区，固定大小，FIFO
Think buffer：近端思维工作区，固定大小，溢出后压缩总结
External episodic memory / recall archive：无限长外部回忆库，存整个历史视频的 segment embedding、keyframe、caption/OCR/ASR、时间戳。recall 实际上查的是这一层

2、动作协议
V1 不要一上来做任意长度 agent loop。先限制成每个 chunk 最多一次 recall，这样训练和时延都可控。

建议统一输出语法：

<think>...</think><action>silent</action>

<think>...</think><action>response</action><response>...</response>

<think>...</think><action>recall</action>
<query>{"query":"...","time_bias":"past_recent|past_far|current","target":"entity|event|ocr|procedure|cause","topk":3}</query>

如果触发 recall：
<think>...</think><action>recall</action>
<query>{"query":"...","time_bias":"past_recent|past_far|any","target":"entity|action|event|ocr|dialogue|procedure|cause","topk":3}</query>

assistant: <think>...</think><action>recall</action><query>...</query>
tool: retrieved segments / keyframes / summaries
assistant: <think>...</think><action>silent|response</action>[<response>...</response>]


system prompt：
你是流视频问答 agent。

每次 assistant turn 必须严格输出以下三种形式之一：
1) <think>...</think><action>silent</action>
2) <think>...</think><action>response</action><response>...</response>
3) <think>...</think><action>recall</action><query>{"query":"...","time_bias":"...","target":"...","topk":3}</query>

规则：
- 每个 turn 只允许一个 action。
- think 只写当前新增判断，不能复述整段视频。
- 若当前 recent window 证据已足够，直接 response，不要 recall。
- 若问题依赖已经离开 recent window 的历史内容，优先 recall。
- recall query 必须短、可检索、非完整问句、避免代词和泛词。
- 收到 <recall_result> 后，只能再输出 silent 或 response。
- 若当前 chunk 不该说话，输出 silent。

3、
建议存储窗口 24–32 秒，但注意：“存储窗口”不等于“每次都喂给模型的窗口”。Qwen3-VL 官方训练示例里给了 VIDEO_MAX_TOKEN_NUM=128、FPS_MAX_FRAMES=16，这意味着你不能每步把 32 秒 × 2 fps 的 64 帧都塞进去；更合理的是外部原始缓存保留 24–32 秒，而每次模型前向只选 12–20 个 visual slots
我建议这样选帧：
8 张：最近窗口内均匀采样；
4 张：高运动/场景切换/显著 OCR 帧；
4 张：最新 2–4 秒强保留；
recall 触发时，额外附加4–6个给回忆帧

B. Think buffer

C. External recall archive
这是无限流的关键。建议按 4 秒 segment、2 秒 overlap、2 fps 建库。每个 segment 存：

segment_id / t0 / t1
1–2 张 keyframe
一个短 caption / OCR / ASR 摘要
visual embedding
text embedding
实体/动作/场景标签
salience score

老数据冷存即可，GPU 只在 recall 命中后把 keyframe/mini-clip 拉回。


4、参数配置：
默认 chunk：2s；
活跃问答/高运动/trigger armed 时，降到 1s；
平均 think：24–40 token / 2s chunk；
burst think：64–80 token，只在以下情况启用：用户提问、歧义、recall 前后、因果链更新；
response：
简短即时答：40–80 token；
明确解释型答复：80–140 token；
连续解说：15–40 token；
recall query：8–20 token 主体 + 少量结构字段。
recall 频率建议
训练集 chunk-level action 分布：
<silent> 58–65%，<response> 23–30%，<recall> 10–15%；
线上目标：
总体 recall 5–12%，活跃回溯问答场景 15–25%；

5、什么任务需要 think，什么任务需要 think+recall
5.1 只需要 think，不需要 recall

这些都应该在 recent visual buffer + think memory 内解决：

当前感知：现在画面里是什么、谁在做什么、OCR 当前文本；
短时时序：过去 2–16 秒内动作顺序、是否刚刚发生变化；
trigger 监测：用户说“看到 XX 就提醒我”；
连续解说：直播点评、过程播报；
根据用户水平改写答案：新手版/专家版，只改表达，不改证据。
5.2 需要 think+recall

这些是 recall 的主战场：

超出 recent buffer 的细节回忆；
跨长间隔的人/物 re-identification；
现在与过去的比较题；
长程因果链：先前发生的原因导致了现在的结果；
程序/流程类回溯：前面做到第几步、哪一步错了；
多轮对话引用先前答案；
OCR/数字的历史比对；
需要从“长时间沉默”后突然回答的 delayed response。
5.3 recall 的触发信号

不建议完全靠生成式自由发挥，建议再加一个轻量 action head 或规则分数，辅助校准 {silent,response,recall}。
可用四个触发分量：

temporal_gap_score：问题目标是否明显指向过去；
reference_score：是否出现“刚才/前面/之前那个/上一次”等回指；
uncertainty_score：当前证据不足；
causal_need_score：当前现象解释需要 antecedent。

如果总分高且不在 recall 冷却中，就走 recall。

7、Streamo-Instruct-465K重构：
保留
原视频；
query；
原有 response 的时间区间/时间点；
同一视频的多 response 关系
不直接保留
原 response 的表面文本，只把它当弱标签；

重构
最早可答时刻 t* 重估

10、数据构造流程
A. 8 卡 AMD 上的 qwen3.5 397B FP8
负责文本层高价值工作：
任务生成；
稀疏 think 轨迹生成；
因果链补全；
response 改写；
disputed sample 仲裁。

B. 8 卡 H20
小/中型 VL 模型批量预处理：
dense caption
entity/action/OCR/ASR 摘要
easy QA
recall query 生成与检索验证
主训练：SFT + RL
检索工具：Embedding / Reranker 批推理

Stage 0：便宜预处理
切 chunk / scene；
2 fps 抽帧；
OCR / ASR；
小 VL 模型做 dense caption；
为每个 4 秒 segment 建 embedding、文本摘要、实体动作标签

Stage 1：生成“结构化时间线”
输出不是自然语言长文，而是：
事件表；
实体轨迹；
过程状态；
OCR 时间线；
ASR 时间线；
候选 support spans。

Stage 2：397B 一次生成多任务包
一条 60–120 秒的结构化时间线，397B 一次输出 8–20 个候选任务，每个任务只输出：
任务类型；
ask_time；
answer_time；
support_spans；
need_recall；
sparse think milestones；
canonical_answer；
natural_response。

Stage 3：用脚本或小模型把 sparse trace 扩成 dense chunk trace
不要让 397B 为每个 chunk 写 think。
大模型只写“状态变化点”和“关键回合”，例如：
t=12：开始怀疑在煮面；
t=28：看到下锅，确认；
t=41：用户追问；
t=42：需回忆前面拿了什么调料。
然后再自动展开成 chunk-level <silent>/<response>/<recall>。

不要把“简单题”全部删掉。
一个 streaming assistant 必须有 25–30% easy fast-path 数据，否则它学不会“当前就答、别想太多”。

我建议按四档难度分桶：
Easy：4B/7B 无 recall、只看 recent buffer 就能稳定答；
Medium：需要 think，但不需要 recall；
Hard：需要 recall，no-recall 会失败；
Very hard：长程因果 / 多轮 / 多跳，只有 teacher 或多次采样能做稳。

训练集建议保留比例：
Easy 25%
Medium 35%
Hard 30%
Very hard 10%

优先做可验证样本
multiple choice
yes/no
counting
event trigger
canonical slot filling

对开放式答案做双层标注
canonical_answer 用于自动评测；
natural_response 用于 SFT 学表达。

自动过滤规则
Prefix causality check：答案不能依赖未来帧。
Early-silence contrastive check：同题在 t*-1 必须答不出。
Answer stability check：从 t* 开始连续 2–3 个 prefix 归一化答案一致。
Recall hit@k check：若 need_recall=true，检索 top-k 必须覆盖 support span。
Counterfactual removal：去掉 gold support 后，小模型应明显退化。
Dual-small-model agreement：两个小模型/两个解码种子一致才收。
397B only-on-dispute：只有争议样本再送大模型裁决。
这样可以把大模型 judge 压到 10–15% 以内。

四、recall 相关数据构造：这是整个方案的重点

下面这部分最关键。我按 “怎么选视频 → 怎么派生 recall 任务 → 怎么保证 query 能打中 → 怎么自动验” 来讲。

4.1 哪些视频值得进 recall pool

不是所有视频都该做 recall。建议先给每个视频打一个 recallability_score：

recallability_score =
0.25 * duration_score
+ 0.20 * entity_recurrence
+ 0.20 * event_chain_depth
+ 0.15 * procedurality
+ 0.10 * ocr_density
+ 0.10 * dialogue_density
- 0.20 * cut_rate

经验阈值：

duration_sec >= recent_window_sec + 8
event_chain_depth >= 3
entity_recurrence >= 0.3 或 procedurality >= 0.5
cut_rate 太高的剪辑短视频直接降权

最适合 recall 的视频类型：

教程 / 做菜 / 维修 / 装配
监控 / 观察 / vlog 长镜头
剧情 / 综艺 / 长场景对话
屏幕录制 / UI / OCR 连续操作
运动 / 户外里有明显“先前事件导致当前结果”的视频

4.3 从 Streamo 派生 recall-positive 样本

Streamo 的原始价值不在 response 文字，而在 多任务 + 时间区间 + backward / TSQA 语义。论文里明确写了它覆盖 real-time narration、caption、temporal grounding、time-sensitive QA，并对每个视频给了统一时间标注；benchmark 还显式区分了 forward / backward grounding。

具体派生方法：

A. 先找最早可答时刻 t*

对每个 Streamo 样本，用 support interval 当证据锚点，不直接拿原 start/end 当回答时刻。
做 prefix sufficiency search：

def search_earliest_answerable(question, support, prefixes):
    # prefixes: [0:t1], [0:t2], ...
    # 用小VL模型或已训练student判断 canonical answer 是否稳定答对
    # 返回最早稳定答对的时刻 t*

判定标准：

在 t* 时 canonical answer 首次答对
在 t* + 1chunk、t* + 2chunk 仍稳定答对
t* - 1chunk 不能稳定答对
B. 再派生 recall ask time

从 t* 往后取多个 ask_time_ms：

t* + 8s
t* + 16s
t* + 32s
t* + 64s

保留那些满足：

support_segment_ids 与 recent window 无重叠
当前问题仍然有意义
no-recall baseline 失败
gold recall + retrieval 通过
C. 把一条 Streamo 样本拆成 3 条

每个 recall-positive，都配套生成两个 control：

recall-positive
support 已离开 recent window，need_recall=true
no-recall control
同一问题提前到 support 还在 recent window 时问，need_recall=false
false-recall negative
问题表面像“回忆”，但当前窗口已足够，正确 action 应是 response 而不是 recall

这 3 条必须绑在一起进训练，才能把 该 recall 时 recall，不该 recall 时不 recall 训出来。

4.4 从 ThinkStream 派生 recall-positive 样本

ThinkStream 的优点不是长视频，而是 incremental think 的风格非常对。论文核心也是短 reasoning update + 决定何时响应。

做法：

A. 原样保留它的增量 think 风格

这部分直接拿来做 Stage-1 协议 warm start。

B. 人为缩小 recent window，强行制造 recall

例如一段 10s 视频：

训练时 recent window 只给 4s
在 9s 位置问 1s 处发生的细节
这样就能把短视频转成 recall 样本
C. recall 样本只让 teacher 补最小量内容

不要重新生成整条 think 链。
直接复用原始 incremental think，只让 teacher 生成：

need_recall
gold_query
recall 后第二段 think
final response

这样能大幅减少 397B 调用次数。

4.5 recall query 怎么造，才能既短又能打中

这是 recall 数据构造里最容易翻车的点。

query 生成规则

query 必须满足 5 条：

不是问句
不用代词
必须有区分性锚点
尽量不泄露完整答案表述
长度短，目标 10–30 token

我建议 query 结构固定成：

[anchor entity] + [distinct action/state] + [object/attribute] + [optional temporal hint]

例子：

人物回忆：红衣 男人 把 杯子 放到 水槽旁
程序步骤：蒜末 下锅 翻炒 第一步
OCR 历史：白色 价签 19.9 元
因果回忆：杯子 打翻 之前 手碰 桌边
多轮 follow-up：刚才 提到 的 黑色 包 放置位置
query 生成推荐流程

不是一次生成一个 query，而是一次生成 3 个候选：

{
  "candidates": [
    {
      "query": "蒜末 下锅 翻炒 第一步",
      "time_bias": "past_far",
      "target": "procedure"
    },
    {
      "query": "蒜末 倒入 锅中 爆香",
      "time_bias": "past_far",
      "target": "action"
    },
    {
      "query": "先把蒜末下锅 调料顺序",
      "time_bias": "past_far",
      "target": "procedure"
    }
  ]
}

然后自动做：

embedding 检索 top20
rerank 到 top5
看 top3 是否覆盖 gold support
选最短且命中的 query

如果 3 个都不命中：

先用小模型做 query repair
还不行再送 397B
query repair 只改两类东西
加锚点：实体 / 数字 / OCR / 物体属性
换表述：动词同义替换、去掉抽象词

不要把 query 越修越长。
超长 query 往往会把检索带偏。

4.6 recall hard negatives 怎么做

如果没有 hard negatives，retriever 和 reranker 都会看起来很准，但上线一塌糊涂。

每个 recall-positive 至少配 4 类 negative：

1. temporal near-miss

同视频、同场景、前后相邻，但不是 support。

例：

gold: 40–44s 蒜末下锅
neg: 36–40s 切蒜
neg: 48–52s 下洋葱
2. semantic confounder

同实体，不同动作；或同动作，不同对象。

例：

gold: 男人把红杯放水槽旁
neg: 男人把蓝杯放桌上
3. OCR confounder

文本相似，但数字/关键词不同。

例：

gold: 19.9
neg: 18.9
4. same-video random hard

同一视频里风格相近、但答案无关的片段。

推荐比例：

temporal near-miss : semantic confounder : OCR confounder/random
= 2 : 1 : 1
4.7 recall-positive 的自动验收门槛

这是你减少人工的关键。每条 recall-positive 必须过 6 个闸门：

闸门 1：support 真在窗口外
overlap(recent_window, support_span) == 0
闸门 2：retrieval hit
hit@3 == 1
闸门 3：support coverage
max_temporal_overlap(retrieved, gold_support) >= 0.5
闸门 4：no-recall fail

只给 recent window，让小 VL/student 尝试答：

EM/F1 < 0.4  或  slot_match == false
闸门 5：with-recall pass

给 gold query 或 selected query 的 recall 结果后再答：

EM/F1 >= 0.8  或  slot_match == true
闸门 6：counterfactual fail

把 gold support 从 recall_result 里删掉，重新答一次，性能必须明显下降。

score_drop >= 0.5

只有全过，才收进 episode_recall.jsonl。

4.8 recall 数据的样本配比

首版建议 recall 只在后两阶段拉高，不要一开始就塞满。

recall pool 内部类型配比

建议：

retrospective detail：35%
procedural state：20%
compare past vs present：15%
long causal：15%
OCR history：10%
multi-turn follow-up：5%
SFT chunk/action 层面的比例

建议最终成品里：

<silent>：58–65%
<response>：23–30%
<recall>：10–15%

这个比例能把“别乱说、别乱检索”训出来。
