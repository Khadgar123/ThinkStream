第一，信息损失最少的窗口不一定是最应该压缩的窗口。例如某个窗口信息很少，压缩它确实损失很小，但压缩收益也可能很小；另一个窗口信息较多，但结构完整、容易总结，压缩后能省更多 token。最终目标不是“损失最少”，而是：

在 token 预算约束下，最大化未来可回答性 / 最小化未来记忆 regret。

更合适的 teacher score 应该类似：

score(window) =
  + importance_lost_after_summary
  + pending_question_overlap_penalty
  + event_boundary_break_penalty
  + retrieval_future_cost
  - token_saving_gain
  - summary_reconstructability_bonus

也就是说，不只看“丢了多少信息”，还要看“省了多少空间”“这段是否容易压缩”“是否破坏事件边界”“是否和 pending question 相关”。
可以保留 greedy teacher 作为第一版，但最好记录一个指标：

compression_regret = 当前 greedy 选择后，未来 N 步 QA / memory score 的下降

用这个指标筛掉明显短视的 gold window。

3. Gold range 可能不唯一，SFT 到单一窗口会惩罚合理选择

很多时候有多个窗口都可以压缩。例如：

A: [20-28] 普通走路
B: [24-32] 普通走路 + 开门
C: [28-36] 开门 + 进入房间

teacher 选 B，但模型选 A 或 C 也可能完全合理。普通 token-level SFT 会把 A/C 当错，导致模型学到的是 teacher 的偶然偏好，而不是“好压缩窗口”的原则。

建议 C2 不要只存一个 gold range，而是存：

{
  "best": [24, 32],
  "acceptable": [[20, 28], [28, 36]],
  "bad": [[32, 44]],
  "reason": {
    "best": "完整事件边界，信息密度适中",
    "bad": "包含 pending question 相关观察"
  }
}

训练时可以：

SFT 只用 best；
评估时 top-k range 都算合理；
或者加一个 pairwise ranking loss：best > acceptable > bad；
或者生成多个等价样本，避免单一 gold 过拟合。

这对“模型自己指定窗口”非常重要。

4. Teacher 选窗口时不能使用学生不可见的信息

teacher 可以用 teacher_caption 做验证，但gold range 的选择原则必须能从学生可见状态中推断出来。否则会出现 hidden-label 问题：teacher 因为知道某个细节很重要，所以不压缩某段；但学生的 recent_thinks 里根本没有这个细节，学生无法学会这个选择。

建议拆成两层：

teacher_caption：用于评估窗口压缩后真实损失
student_visible_memory：用于生成可学习的 range label

换句话说，teacher 最终可以检查这个 label 是否好，但它的决策输入最好限制为：

compressed_segments + recent_thinks + pending_questions + 当前视觉窗口

不要让 teacher 因为隐藏 evidence graph 里的事实，生成学生输入中不可推断的窗口偏好。

5. Compression summary 必须明确“只能压缩文本记忆”，否则视觉窗口会泄漏

你的设计里 recent_thinks 和 visual_window 有重叠，这是合理的，因为文本记忆和视觉窗口是互补的。

但压缩训练有一个隐性风险：模型在 compress turn 里同时看到 recent_thinks 和视觉帧。如果被压缩的时间段仍在 visual_window 里，模型可能从视觉帧中补充很多 recent_thinks 没写过的细节。这样 summary 表面上是在压缩文本记忆，实际上变成了“重新看视频写摘要”。

这会破坏你的三层隔离目标：

student_think → compressed_summary

可能变成：

student_think + 当前视觉帧隐含细节 → compressed_summary

建议明确规则：

方案 A：压缩 summary 只能引用 recent_thinks

验证时做 provenance check：

summary 中的每个实体 / 数量 / OCR / 状态变化
必须能在被压缩的 recent_thinks 中找到支持。
方案 B：如果允许看视觉帧，则要把它定义成“文本+视觉再编码”

那就不再是纯文本记忆压缩，而是 multimodal memory consolidation。这个也可以，但要明确，因为它和后续 recall/QA 的信息流假设不同。

我更建议先用方案 A：compressed_summary 只能来自 student_think。视觉窗口仍然可以在 input 中，但 summary 不能引入 recent_thinks 没有的事实。

6. 需要明确 compress turn 的状态更新顺序

现在协议是每一步输出：

<think>...</think><action>compress</action><summary>...</summary>

但这里有一个运行时歧义：

当前 timestep 的 think 是否先写入 recent_thinks？
压缩范围是否允许包含当前刚生成的 think？
summary 是否可以引用当前 think？
系统是先 append think 再 compress，还是先 compress 再 append think？

建议固定成：

输入 memory_state = t 时刻之前的 memory
模型输出 current_think + compress(range over input recent_thinks only)
系统执行：
  1. 对 input recent_thinks 中指定 range 做替换
  2. 再 append current_think 到 recent_thinks

或者：

系统执行：
  1. append current_think
  2. compress range，包括 current_think 也允许

二者都可以，但必须二选一。为了降低复杂度，我建议第一种：compress range 只允许覆盖 input 里已经存在的 recent_thinks，不覆盖当前 output 的 think。这样数据构造和验证最清晰。

9. 要避免模型学成“永远压最早窗口”

如果 teacher policy 常常选最早的低风险窗口，C2 会退化成 FIFO：

只要触发压缩，就压 recent_thinks 最前面的 4–8 条。

这不一定坏，FIFO 是强 baseline，但它不是“精确记忆管理”。要让模型真的学会选择，需要构造反例：

最早窗口包含 pending question 相关内容 → 不压最早
最早窗口包含 OCR / 数字 / 关键实体 → 不压最早
中间窗口是完整低价值事件 → 压中间
最新窗口虽然低价值，但还在视觉窗口中且事件未结束 → 不压最新

每批 C2 数据都应该统计：

teacher 选择 earliest / middle / latest 的比例

如果 earliest 占比太高，模型会模式坍缩。
10. “压缩后可 recall 原始 thinks/frames”会改变压缩策略

你的系统里 recall 可以访问历史 thinks、compressed summaries、historical frames。 这会带来一个策略问题：

如果模型知道丢掉的细节之后还能 recall 回来，它可能学会更激进地压缩 summary，只保留粗略信息，把细节交给 recall。

这未必错，但要看你的目标。如果目标是：

compressed_summary 本身尽量保留可问细节

那就要增加 QA-after-compress loss，让 summary 直接承担更多答案。

如果目标是：

summary 保留主线，细节通过 recall 找回

那就要显式把 recall cost 纳入 teacher score。否则模型会学到：

能省就省，反正以后 recall。

建议给每个压缩样本做两类评估：

1. summary-only QA accuracy
2. summary + recall QA accuracy

并在 teacher score 里加入：

recall_penalty = 未来需要 recall 才能回答的次数 / 成本
11. DAgger / student rollout 不应该是“后续增强”，而应该尽早进入 C2

因为你的记忆状态是由模型自己生成的 think 和 summary 组成的。训练时如果 memory 都来自 397B teacher rollout，推理时却来自 student 自己，分布偏移会很明显。这个问题和 imitation learning 里的 compounding error 很像：learner 一旦犯错，就会进入专家演示中很少出现的状态；DAgger 的核心动机就是让训练覆盖 learner 自己诱导出来的状态。

你的文档里已经有 DAgger 计划，这点很好。 但我建议把它提前：

C1: teacher memory
C1-D: student/noisy memory 上的 fixed-range compression
C2: teacher memory 上的 range selection
C2-D: student/noisy memory 上的 range selection

也就是不要等到很后面才做 DAgger。否则 C2 学到的 range policy 可能只适用于“高质量 teacher thinks”，不适用于 student 自己写得短、漏、重复的 memory。

这和 scheduled sampling 的动机也一致：训练时总是条件在标准历史上，推理时条件在模型自己生成的历史上，会导致误差累积。

12. compressed_segments 超过 5 后的合并也要训练，否则是隐藏损失源

你文档里说 compressed_segments 超过 5 段时，系统合并最老两段。 这一步如果完全由系统规则做，模型就不会学会“二级压缩”或“摘要的摘要”。

但长期记忆管理里，真正难的是：

think → summary
summary + summary → higher-level summary

建议增加一种样本：

<merge_compress_trigger segments="seg_0,seg_1"/>

输出：

<action>compress</action>
<summary>{"source_segments":["seg_0","seg_1"],"text":"..."}</summary>

否则你会有一级压缩训练，但二级压缩靠系统硬合并，可能造成不可控信息丢失。

13. action 优先级要明确：提问、recall、compress 同时出现怎么办？

协议里每步只能输出一个 action：

silent / response / recall / compress

但运行时可能出现冲突：

recent_thinks 达到 80% 阈值，同时用户提问
pending question 等待触发，同时系统想压缩
recall_result 返回，同时 memory 超阈值

建议定义优先级：

post-recall response > 当前用户 response/recall > pending trigger response > compress > silent

或者：

用户交互动作优先，compress 可以延后一帧。

这会避免训练样本里出现同一状态下有时 response、有时 compress 的冲突标签。

14. 我会这样修改你的两阶段方案

可以改成四个小阶段，而不是硬两阶段：

C1：固定范围压缩

系统给 range：

<compress_trigger range_ids="obs_10:obs_16"/>

模型只学：

当前 think + compress summary

目标：学格式、摘要、压缩后推理。

C2：自选窗口 + summary

系统只给：

<compress_trigger/>

模型输出：

{"time_range":[24,36],"text":"..."}

目标：范围选择和摘要生成联合训练。
