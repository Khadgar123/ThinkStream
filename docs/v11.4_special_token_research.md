# Special-Token SFT Research — Industry Patterns vs ThinkStream

Survey of how 9 reference codebases handle the "model needs to emit a
custom structural tag like `<action>` / `<tool_call>` / `<recall>` in
free generation" problem — relevant to ThinkStream's v11.3 cold-start
bug (model omits `<action>` despite eval/action_acc=0.98 teacher-forced).

## TL;DR

**ThinkStream is the only codebase in the surveyed set that adds
structural tags as new tokenizer-vocabulary entries.** Every other
agent / tool-call SFT framework treats these tags as plain text inside
the assistant message and lets the LM tokenize them as multi-token
sequences (`<` + `action` + `>` = 3 tokens for the natural BPE).

This explains why ThinkStream hit the cold-start bug — and the
industry sidesteps it by never creating new tokens in the first place.

## Why everyone else avoids new tokens

The cold-start magnitude problem ThinkStream observed in v11.3 is the
exact reason. Adding a new token to the tokenizer requires
`model.resize_token_embeddings(...)`, which initializes the new row
to either zero (older transformers) or the mean of existing
embeddings (>=4.40). Either way the new row's magnitude is small
relative to well-trained tokens.

At sampling time, the next-token logit is `dot(hidden_state, embed_t)`.
A small-magnitude `embed_t` produces a small logit even if the
direction is correct. Natural-English alternatives (e.g., the word
"response") have well-trained magnitudes, so the softmax picks them
instead of the structural token.

SFT training does push gradient through the new-token row, but
through a flat region of softmax — gradient is small until magnitude
catches up. One epoch usually isn't enough.

**Industry workaround**: avoid new tokens entirely. Pay the cost of
multi-token tag emission (`<action>` is now 3 BPE tokens instead of 1)
in exchange for free magnitude. Every individual sub-token is a
well-trained existing vocab entry.

## What ThinkStream chose, and why

ThinkStream went the new-token route because:

1. Single-token tags are **faster at inference** (1 forward step
   per tag vs 3-5 with multi-token).
2. **Attention efficiency**: each tag occupies one position, not
   multiple, so the visual-window's effective context is preserved.
3. **Reward function precision**: the RL reward functions in
   `grpo.py` look for token-id matches against `<action>`,
   `</response>`, etc. With multi-token tags, regex matches on
   decoded strings are still fine, but token-level loss masks
   (`SPAN_WEIGHTS`) become significantly more complex (a span starts
   on an opening sub-token sequence, not a single token).

These are real benefits that justified the choice — but they came
with the cold-start risk that v11.4's `smart_init_special_token_embeddings`
fixes.

## Three nearby patterns for reference

### LLaMA-Factory's three init modes

`src/llamafactory/model/model_utils/embedding.py:32-150` —
`init_special_tokens` flag:

1. **`noise_init`** (default): mean-of-existing + Gaussian noise
   `std = 1/√d`. Breaks symmetry so multiple new tokens don't share
   the same embedding row.
2. **`desc_init`**: load a YAML config mapping each new token to a
   semantic description ("A token marking the start of structured
   thinking"). Tokenize the description, average those tokens'
   embeddings, use as the seed. More principled than just-the-word
   approach because the description carries more context.
3. **`desc_init_w_noise`**: combination of (1) and (2).

ThinkStream's `smart_init_special_token_embeddings` is closest to a
simplified `desc_init` — it derives the descriptor from the tag's
inner word (`<action>` → "action") instead of an external YAML.
This is enough for single-word concept tags but might lose semantics
on multi-word tags (`<recall_result>` → mean(recall, result), which
is approximately the union-of-meanings interpretation, not the
joint-meaning).

**Possible v11.5 enhancement**: add an optional `new_special_tokens_config`
YAML mirroring LLaMA-Factory's pattern:
```yaml
"<action>": "structural marker for the model's decision action type"
"<response>": "marker for the text response content within a decision"
"<compress_trigger>": "system signal that memory has reached the budget threshold"
```
Falls back to inner-word averaging when missing. Effort: ~30 LOC,
backward-compatible with current code.

### DeepEyes's RL-time format reward

`recipe/deepeyes/deepeyes.py:316-339` — when free generation produces
malformed output (regex extraction fails on `<tool_call>...</tool_call>`),
the format reward returns -1.0:

```python
if not tool_call_match:
    return -1.0  # explicit format penalty
return 0.0
```

Combined into `final_score = 0.8*acc + 0.2*format + 1.2*tool_use`. The
format component is sparse (only fires on extraction failure) but
cumulative over multiple chunks/turns.

ThinkStream's RL `_compute_format_reward()` (grpo.py:337-342) does
something similar at the chunk level (proportion-of-chunks-matching
regex), with weight 0.09 in DEFAULT_REWARD_WEIGHTS. So this pattern is
already adopted at RL stage — but **no SFT-time format-verification
loss** exists.

**Why it matters less for ThinkStream than DeepEyes**: DeepEyes's
tags are multi-token (text in content), so format failures are
multi-step. ThinkStream's tags are single-token, so once smart_init
gives them adequate magnitude, the per-token CE loss already covers
the right thing — no additional format loss should be needed.

### ReMemR1's `{% generation %}` chat template marker

ReMemR1 uses a Qwen2-style Jinja chat template that wraps generative
content in `{% generation %} ... {% endgeneration %}` blocks. This
tells the tokenizer which token positions are "model output" (loss
applies) vs "context only" (loss masked). Cleaner than ThinkStream's
manual ans_start/ans_end pattern (`data_processor.py:786-811`).

Not a fix for ThinkStream's specific bug, but a **cleaner abstraction**
that v11.5+ could migrate to. Low priority unless you hit chat-template
drift bugs.

## Decision: stay with new tokens, just init them right

The smart_init fix in v11.4 directly addresses the magnitude problem.
The survey shows we're the only codebase taking this approach, but
**that's because we're the only codebase needing single-token
structural tags for inference efficiency**. The choice is valid;
v11.4's init strategy makes it work.

What we don't need to add:

- **Multi-token migration** (drop new tokens entirely): would require
  re-running pass2 + pass3, retraining batch1 SFT, invalidating all
  existing infrastructure. Way too costly for a problem already
  solved by smart_init.
- **SFT-time format-verification loss**: if smart_init works (high
  confidence — magnitude is now in normal range from step 0), the
  per-token CE loss already trains the right thing. Format reward
  at RL is enough.
- **Higher learning rate on new-token rows**: not a pattern anyone
  uses. Would risk destabilizing existing well-trained tokens via
  shared optimizer state. Skip.

What's worth considering for v11.5:

- **YAML descriptor config** (LLaMA-Factory pattern, ~30 LOC):
  for tags whose inner word doesn't capture the concept well
  (e.g., `<compress_trigger>` is more "memory pressure signal" than
  literally "compress" + "trigger"), allowing a human-written
  description gives a richer init.
- **Post-init noise injection** (~5 LOC): break per-token symmetry so
  multiple new tokens with similar inner-words (`<query>` + `<queries>`)
  don't start from identical seeds. Cheap insurance against subtle
  symmetry-breaking failures.

## Verification plan for v11.4

After running the next SFT (with smart_init enabled), check these
wandb metrics ordered by importance:

1. **`eval/format_compliance` from step 50 onward**: should be > 0.9.
   This is the closed-book check ("does the model decide to emit
   `<action>` after `</think>`"). If < 0.5, smart_init didn't work
   and we need to investigate (run
   `scripts/audit/diagnose_action_tag_omission.py`).
2. **`eval/action_acc` should still be ≥ 0.95**: smart_init shouldn't
   regress the open-book metric.
3. **`eval/format_compliance_silent` may be N/A**: silent samples
   don't have `<action>` tags (silent IS the action keyword via the
   silent-action-after-action sequence — actually depending on which
   format is used for silent samples). Check that silent eval doesn't
   spuriously fail format compliance.
4. **`eval/response_argmax_acc_response`**: should be ≥ 0.85 after
   1 epoch. Tells us answer-content quality is high.

If `format_compliance < 0.5` and `action_acc > 0.9`, the cold-start
bug persists despite smart_init — then escalate to:
- Run the diagnostic script to verify embedding magnitude post-init
- Check that `register_special_tokens` actually fires (wasn't already-
  in-vocab fallback path)
- Consider YAML descriptor config (LLaMA-Factory pattern)
