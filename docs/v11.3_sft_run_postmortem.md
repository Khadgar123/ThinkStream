# v11.3 SFT Run Postmortem — silent over-correction

First production SFT run with the v11.3 weight rebalance (`compress 0.8→2.5`,
`smoothing 0.7→1.0`, `think span 0.3→0.6`, `no_query silent 0.5→1.0`)
showed a clean compress-collapse fix but a new failure mode:
**silent action accuracy degrades on eval as training progresses**.
This doc records the diagnosis, the recommended response, and the
weight settings to use if a fresh run is needed.

## Run setup

- 1 epoch ≈ 310 steps
- Eval every ~50 steps
- Checkpoint: v11.3 weights (commit `35eddb0`) + token-driven compress
  (commit `5814e23`) + per-class wandb metrics (commit `43316f0`)

```
train/action_argmax_acc_silent: 0.125 → ~0.9   ← never reaches 1.0
```

## Diagnosis: not classical overfit, it's class-rebalance over-correction

eval_loss rise is **entirely** explained by silent_acc dropping 13
percentage points (99% → 86%). All other action classes saturated at
1.0 on eval — they have nothing left to gain. Train loss says the
model is fitting, but the gradient signal it's fitting *on* skews
heavily toward non-silent classes.

### Quantitative gradient share, v11.3 settings

`class_balance_smoothing = 1.0` produces:

```
silent base freq   = 70%   →  inv_freq = N/(4 × 0.7N) = 0.357
compress base freq = 25%   →  inv_freq = N/(4 × 0.25N) = 1.000
recall  base freq  =  4%   →  inv_freq = N/(4 × 0.04N) = 6.250
response base freq =  3%   →  inv_freq = N/(4 × 0.03N) = 8.333

normalized (mean=1):
  silent    ≈ 0.17×     (silent gets ~6× LESS sampling than uniform)
  compress  ≈ 0.49×
  recall    ≈ 3.05×
  response  ≈ 4.07×
```

`ACTION_WEIGHTS`:

```
silent          1.0
compress        2.5
response        1.5
recall_query    1.5
```

Effective per-update gradient share = sampling_weight × action_weight:

```
silent    : 0.17 × 1.0 = 0.17
compress  : 0.49 × 2.5 = 1.23     ← 7.2× silent
response  : 4.07 × 1.5 = 6.10
recall_q  : 3.05 × 1.5 = 4.58
```

**compress is pulled 7.2× harder than silent**, response 36× harder.
v11.2 had the opposite imbalance (silent crowded out compress), v11.3
swung the lever to the other extreme. The training data confirms it:

- `train/action_argmax_acc_silent` saturates at ~0.9, **never** 1.0,
  meaning the model can't even fit silent perfectly on the training
  set — silent samples are too rare per gradient step to consolidate.
- All non-silent classes hit 1.0 on both train and eval before
  silent stops degrading. They've extracted everything available;
  continuing to train only takes from silent.

## summary_argmax stuck at 86% is a different (benign) issue

`<summary>{"time_range":[t0,t1], "text":"..."}</summary>` is part copy
(time_range from trigger) and part generation (description text).
Token-level argmax acc on the GENERATED text part has natural variance
— exact-token match isn't a meaningful target for a multi-word
description. 86% mixed acc is consistent with ~98% on the time_range
copy + ~75% on the text. This is **not** a training failure, it's a
metric resolution limit. Don't tune for this number above 90%.

## Recommended response — two paths

### Path A: use ckpt-150 directly (recommended)

The eval_loss minimum at step 150 (epoch 0.48) is the best snapshot
this run will produce. Both compress (1.0 acc) and silent (~0.95 acc)
are healthy there. Continuing past step 150 trades silent for nothing.

**Action**: take ckpt-150 as the SFT seed, proceed directly to RL.
RL's advantage signal works in the policy space, not token loss space
— small per-class action probability errors get repaired naturally
through the reward gradient (silent_quality reward at w=0.18 directly
pushes silent up when correct).

This is the **conservative path**. No retraining, no fresh
data construction. Get to RL ASAP and validate end-to-end.

**Watch for** (during RL): if `train/action_argmax_acc_silent` in RL
audit logs continues to decline, fall back to Path B.

### Path B: rebalance and retrain (if Path A fails or you have GPU budget)

Pull the lever back from "aggressive rebalance" to a moderate setting:

```python
# thinkstream/sft/data_processor.py:70-92
ACTION_WEIGHTS = {
    "response":              1.5,    # unchanged
    "silent":                1.2,    # 1.0 → 1.2 (boost silent)
    "compress":              1.8,    # 2.5 → 1.8 (still > silent, no longer 7× crush)
    "merge_compress":        1.8,    # 2.5 → 1.8 (mirror compress)
    "recall_query":          1.5,    # unchanged from v11.3
    "recall_response":       1.5,    # unchanged from v11.3
    "recall_silent":         1.0,    # unchanged from v11.3
    "proactive_recall_query":1.5,    # unchanged
    "proactive_recall_silent":1.0,   # unchanged
}

# thinkstream/sft/argument.py:102
class_balance_smoothing: float = field(default=0.85)   # 1.0 → 0.85
# (not back to 0.7 either — that was the v11.2 setting that
#  caused compress collapse. 0.85 is the midpoint.)
```

Effective gradient ratio under Path B settings:

```
silent_sampling   ≈ 0.357^0.85 ≈ 0.42  → norm 0.27
compress_sampling ≈ 1.000^0.85 ≈ 1.00  → norm 0.65
recall_sampling   ≈ 6.250^0.85 ≈ 4.62  → norm 3.00
response_sampling ≈ 8.333^0.85 ≈ 5.95  → norm 3.86

silent grad   : 0.27 × 1.2 = 0.32
compress grad : 0.65 × 1.8 = 1.17     ← 3.6× silent (was 7.2×)
response grad : 3.86 × 1.5 = 5.79
```

silent vs compress drops from 7.2× to 3.6× — strong but not crushing.

**Mandatory if Path B**: add early-stopping on
`eval/action_acc_silent`. Stop training when it drops below 0.95 or
when it has decreased for 2 consecutive evals. Without this, Path B
can still over-correct just at a different epoch.

```python
# scripts/sft_per_timestep.sh — add the relevant HF Trainer args, or
# wire a custom EarlyStoppingCallback that watches eval/action_acc_silent
--metric_for_best_model "eval/action_acc_silent"
--greater_is_better True
--load_best_model_at_end True
--early_stopping_patience 2          # stop after 2 evals worse than best
--early_stopping_threshold 0.005     # (smoothing for noise)
```

The HF Trainer doesn't natively let you choose a custom metric like
`eval/action_acc_silent` for early stopping — you may need a small
`EarlyStoppingCallback` subclass that reads this key from `metrics`.

## What NOT to do

1. **Don't roll smoothing back to 0.7**: that was v11.2 (which had
   compress collapse). 0.85 is the midpoint, calibrated against the
   measured gradient ratios above.
2. **Don't push silent ACTION_WEIGHT to 2.0+**: same lever, opposite
   direction; you'll re-create the v11.2 silent-dominance bug.
3. **Don't continue training past step 150 hoping silent will
   recover**: it won't. compress/response have hit 1.0 on eval, the
   only thing left for the optimizer to take from is silent.
4. **Don't tune for summary_argmax > 90%**: it's a generation task
   with natural variance, not a copy task. The cap is structural.

## Decision criterion for choosing A vs B

- **Path A** if: you trust ckpt-150's silent ≥ 0.95, you have RL
  capacity ready, you want to validate the full pipeline E2E.
- **Path B** if: you have GPU budget for one more SFT run, you've
  prepared an early-stopping callback that watches silent_acc, and
  RL infrastructure isn't ready yet.

Default recommendation: **Path A**. RL > extra SFT epochs at this
stage of the project.

## Open question (deferred to v11.5)

The v11.3 audit identified that the underlying instability comes from
**coupled levers**: ACTION_WEIGHTS multiplies the per-token loss while
class_balance_smoothing multiplies the sampling rate. The two are not
algebraically independent — moving one changes the optimal value of
the other. v11.5 should investigate:

1. Decoupling: e.g., set ACTION_WEIGHTS=1.0 uniformly (only sampling
   does class balance) OR set smoothing=0.0 (uniform sampling, only
   ACTION_WEIGHTS does balance). One lever, no interaction.
2. Auxiliary action classification head (MMDuet/ViSpeak/Dispider
   pattern): a separate binary head over the action keyword position
   trained with focal loss or weighted CE. Decouples action decision
   boundary from token-level LM loss; observed in those papers to
   converge to action_acc=1.0 for ALL classes simultaneously without
   the reweighting tradeoff.

These are architectural changes; not in scope for v11.3/v11.4
recovery. Document captures the issue for the next iteration.
