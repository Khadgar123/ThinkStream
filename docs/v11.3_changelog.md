# ThinkStream v11.3 Changelog

Working changelog for changes made on top of v11.2 (which corresponds to
`docs/design.md` v9.5). Organized by stage. File:line citations point at
the exact change locations.

The trigger for this revision was an eval audit on the v11.2 SFT
checkpoint that surfaced three pathologies: compress action collapse
(model never emits `<action>compress</action>` despite system trigger),
response leak at silent chunks (model answers when no question is
pending), and 280-300 token repetitive `<think>` outputs (training cap
was 130). Root-cause analysis traced each to specific knob settings;
this revision fixes the knobs and adds the monitoring needed to catch
the same failure mode early next time.

## Stage 1 â€” Data construction

No code changes in this revision. Batch1 output (320 videos, 24,425
thinks, 4,202 compression events) was QC'd and pushed to git
(`data/agent_v5/rollout/`, commit a8639f7) so SFT inputs are
reproducible from the repo alone.

QC findings:
- 311/319 videos with frames present (97.5%); 8 zero-chunk videos
  (`l5WUzgU_65g`, `RR-ZEO4r1c8`, `5QvsiIHl3m0`, `ef2AEHGFUbU`,
  `Zws_MdZS4_0`, `ECO1qNKfjOM`, `bhJwNnHNkRw`, `afoIbXR4-2g`) â€” these
  produce zero downstream samples but a defensive filter was added at
  SFT load time as belt-and-suspenders (see Stage 2).
- Think token length: mean 73, p99 92, max 105 â€” all within the 130-tok
  verify cap. Quality verified frame-by-frame on 5 sample videos
  (text overlays, transitions, static-scene detection all accurate;
  one OCR error: "LIKE" hallucinated for "SUBSCRIBED" button).
- Compress range coverage: 86.6% of summary `[t-t]` markers fall fully
  inside the declared `time_range` (additional 7.9% partial; remaining
  5.5% use prose time expressions like "Between Xs and Ys" that the
  regex doesn't catch).

### Pass3 LLM call optimization (`scripts/agent_data_v5/{pass3a_cards,pass3b_placement,pass3c_samples}.py`)

5 of 7 LLM call sites switched from `thinking=True` to
`thinking=False`. Reasoning: empirically the teacher's thinking budget
went unused (output lengths far below `max_tokens`), and the tasks
(verification, classification, templating, keyword extraction) don't
benefit from chain-of-thought.

Sites changed:
| Call site | Purpose | Reasoning complexity |
|---|---|---|
| `pass3a_cards.py:_verify_one_card` | binary classifier (card valid/invalid) | 2/5 |
| `pass3b_placement.py:_check_visibility_one` | yes/no answerable check | 3/5 |
| `pass3c_samples.py:_generate_response` | SFT label for descriptive/short_exact (70% of calls already skip via canonical_answer fast-path) | 2-3/5 |
| `pass3c_samples.py:_generate_recall_query` | 3-5 keyword extraction with optional time_range | 3/5 |
| `pass3c_samples.py:_generate_recall_think` | 3-way template branching (historical_frames / distractor / failure) | 2/5 |

Sites kept thinking (genuinely complex reasoning):
- `pass3a_cards.py:generate_cards_for_family` â€” multi-family multi-constraint, distractors must come from video, strict format rules.
- `pass3c_samples.py:_generate_fork_think` â€” answer-leakage avoidance: must mention active question without leaking the answer (without CoT, Pass4 leakage check rejects ~40% in early A/B).

Implementation: each call now reads a dedicated PASS_CONFIG sub-key
(`pass3a_verify`, `pass3b_visibility`, `pass3c_response`,
`pass3c_recall_query`, `pass3c_recall_think`, `pass3c_fork_think`)
and passes `enable_thinking=cfg["thinking"]` to `client._call_one`.
The umbrella `pass3c` entry is kept as a legacy fallback for any
external caller. `max_tokens` left at 16384 throughout: GPU has
headroom for 16K @ 1024 concurrent and the speedup comes from
disabling reasoning, not from cap reduction.

Estimated speedup at the pass3 layer: ~3.5Ă— per video (most heavily
on `pass3b_visibility`, which fires per-card-per-chunk).

### Token-driven compress range (`scripts/agent_data_v5/config.py`, `model/agent_loop.py`, `eval/streaming_vllm.py`)

v11.2 had a policy split: `pass2_rollout` enumerated variable ranges
in `[COMPRESS_RANGE_MIN, COMPRESS_RANGE_MAX] = [4, 8]` and picked via
`score_range_for_compression`, but `agent_loop` and `streaming_vllm`
hardcoded "first 4 thinks" for the inference-time compress range.
Inference behavior didn't match training data distribution. v11.3
unifies: all four stages now use the same token-driven helper.

Constants:
- `COMPRESS_RANGE_MAX`: 8 â†’ 12 (handle short-think edge cases where
  4-8 chunks don't sum to enough tokens to relieve memory pressure)
- `SUMMARY_TOKENS_MAX`: 180 â†’ 280 (33% of pass2 summaries hit the
  180 cap; 280 matches the teacher p80 of summary lengths and fits
  8-12-chunk compression densities)
- New `COMPRESS_REMOVE_TOKENS = 350` (target eviction per
  compression; lands post-compress memory near hysteresis 330)

Helper: `model/agent_loop.py:select_compress_range_by_tokens(thinks, count_fn, target=350, min_n=4, max_n=12)`
returns the smallest N >= MIN such that the cumulative tokens of the
first N thinks reaches `target`, capped at MAX. Returns 0 when there
are too few thinks to compress.

Call sites:
- `MemoryState.compress(...)` legacy `compressed_chunks=None` path
  now uses the helper instead of `recent_thinks[4:]`.
- `StreamingAgentLoop.step(...)` builds the compress_trigger range
  from helper-selected thinks (was `[:4]`).
- `streaming_vllm.py:_maybe_compress_trigger` mirrors agent_loop.

Pass2's enumerative scoring is unchanged â€” it's already a
strict-superset of the inference policy, so existing batch1 data
(ranges 4-8) remains a valid subset of the new [4,12] policy space.

## Stage 2 â€” SFT

### Action / span weight rebalance (`sft/data_processor.py:53-92, 124-138`, `sft/argument.py:102-110`)

Diagnosis: v11.2 SFT collapsed because (a) `ACTION_WEIGHTS["compress"] = 0.8`
explicitly downweighted compress with the rationale "timing
non-deterministic, leave to RL" â€” but if SFT never produces compress
action, RL has nothing to optimize on; (b) `class_balance_smoothing=0.7`
delivered only ~2.6Ă— boost to compress (25% base freq), insufficient
to compete with silent's 70% prior; (c) `SPAN_WEIGHTS["think"] = 0.3`
made think tokens contribute ~15% of total token loss â€” the model
never learned the 130-tok cap and collapsed to verbose boilerplate
at inference; (d) the "no_query" branch in question_window/patrol/
legacy roles was 0.3-0.5, explicitly under-weighting "no question
pending â†’ silent" examples so the model learned silence is "less
important" without an active query.

Fixes:
- `ACTION_WEIGHTS["compress"]`:        0.8 â†’ **2.5**
- `ACTION_WEIGHTS["merge_compress"]`:  0.8 â†’ **2.5**
- `ACTION_WEIGHTS["recall_query"]`:    0.8 â†’ **1.5**
- `ACTION_WEIGHTS["recall_response"]`: 1.0 â†’ **1.5**
- `ACTION_WEIGHTS["recall_silent"]`:   0.8 â†’ **1.0**
- `ACTION_WEIGHTS["proactive_recall_query"]`:  0.8 â†’ **1.5**
- `ACTION_WEIGHTS["proactive_recall_silent"]`: 0.8 â†’ **1.0**
- `SPAN_WEIGHTS["think"]`:             0.3 â†’ **0.6**
- `class_balance_smoothing` default:   0.7 â†’ **1.0**
- `_get_sample_weight()` `question_window` no_query branch: 0.5 â†’ **1.0**
- `_get_sample_weight()` `patrol` no_query branch:          0.3 â†’ **0.8**
- `_get_sample_weight()` legacy no_query branch:            0.3 â†’ **0.8**

### 0-chunk video filter (`sft/data_processor.py:PerTimestepDataset.__init__`)

Defensive empty-sample filter: drops samples missing
`input.system` / `input.visual_window` / `output`, OR samples whose
`visual_window.frames == 0`. The 8 zero-chunk videos in batch1 should
already produce zero rows in train.jsonl (pass3c needs thinks to
generate samples), but the filter is cheap and catches future
schema corruption too. Logs `Dropped N empty/corrupted samples`
on rank 0.

### Unique-think-rate weighting (`sft/data_processor.py:PerTimestepDataset`, `ClassBalancedDistributedSampler`, `sft/argument.py`, `sft/trainer.py`)

Each sample is annotated with `_unique_rate = len(set(thinks)) / len(thinks)`
computed over its `input.memory.recent_thinks`. When
`--unique_think_weight True` is passed (default `False`),
`_get_train_sampler` extracts these rates and passes them as
`extra_weights` to `ClassBalancedDistributedSampler`, which multiplies
them into the per-class inv-freq weights post-normalization.

Effect: a sample whose memory snapshot has 50% duplicate thinks gets
half the sampling probability of an otherwise-equivalent unique
sample. Mitigates the static-scene case (~5% videos with >5%
repetition; worst is a yoga session at 28%).

Backward-compat: `extra_weights=None` (default) preserves exact
v11.2 sampler behavior.

### Wandb per-class diagnostics (`sft/trainer.py`, `sft/data_processor.py:_extract_eval_positions`)

Live per-class metrics so the next SFT run surfaces compress collapse
/ response leak / think repetition in wandb without needing eval to
catch them.

Train (every `logging_steps`, via `log()` override):
- `train/loss_by_class_<sample_type>` â€” per-action-type loss; lets
  you see if compress lags silent.
- `train/sw_mean_<sample_type>` â€” mean per-sample weight per class.
  Sanity-check `ClassBalancedSampler` is upweighting compress after
  smoothing=1.0 fix.
- `train/n_frac_<sample_type>` â€” per-batch class composition.
  With the sampler change, compress/recall should stay near-uniform
  across batches.
- `train/action_argmax_acc_<sample_type>` â€” teacher-forced action
  keyword argmax accuracy per class. Leading indicator: compress
  argmax acc <50% at step ~200 means format binding is still broken;
  bump weights more.

Eval (every eval, extended `_eval_acc`):
- `eval/summary_argmax_acc[_<sample_type>]` â€” token-level argmax
  accuracy inside `<summary>...</summary>`. Maps directly to
  "compress range coverage rate" â€” high value means model copies
  the trigger range correctly into JSON `time_range`.
- `eval/query_argmax_acc[_<sample_type>]` â€” token-level argmax
  accuracy inside `<query>...</query>`. Recall query format / content
  quality under teacher forcing.

Position extraction in `_extract_eval_positions`: the function now
also surfaces `summary_span_positions` and `query_span_positions`
(backward-compat: new fields default to []). The all-reduce buffer
in `_all_reduce_eval_acc` was bumped from 4 bands to 8 bands to
cover summary/query match counts.

## Stage 3 â€” Eval

### vLLM-batched offline eval (`eval/vllm_engine.py`, `eval/eval_baseline_vllm.py`, `eval/ovo_bench/eval_ovo_offline_vllm.py`)

The v11.2 eval was sequential (`batch_size=1` `model.generate()` per
sample) and bottlenecked GPU utilization. Phase 1 of the rewrite
adds a vLLM-based offline path that builds all prompts upfront and
runs a single `llm.generate()` over the dataset. Drops torchrun/DDP â€”
vLLM handles tensor parallelism internally.

Key files:
- `vllm_engine.py:91 lines` â€” `LLM` init + `SamplingParams` +
  `prepare_vllm_input` wrapper around `qwen_vl_utils.process_vision_info`,
  matching the canonical pattern in
  `Qwen3-VL/evaluation/VideoMME/run_videomme.py`.
- `eval_baseline_vllm.py:233 lines` â€” `offline_predict_mcq_vllm()`
  reuses `OfflineMCQDataset`, `_load_video_frames`, `parse_answer`,
  `build_results`, `save_results`, `DebugLogger` â€” parsing logic
  zero-changed for parity testability.
- `eval_ovo_offline_vllm.py:92 lines` â€” OVO offline entry, no DDP.

### vLLM-batched streaming eval (`eval/streaming_vllm.py`, `eval/ovo_bench/eval_ovo_streaming_vllm.py`)

Phase 2: chunk-lockstep orchestrator for the agent eval path. At
each `chunk_idx` all live samples build their per-step prompt
(memory + visual_window + queries + user_input), one batched vLLM
call advances every sample, MemoryState updates per-sample. Reuses
`MemoryState` + `build_single_step_messages` from `agent_loop` so
eval prompt format is byte-identical to SFT training. Eval-mode
only (`allow_recall=False`; system-triggered compress).

Key design choices:
- Per-sample state machine (`_SampleRunner`) so different videos
  in the same batch are independent. Done samples (emitted
  `<action>response</action>`) drop out of the live set.
- Default `repetition_penalty=1.1` (CLI flag `--repetition_penalty`):
  the v11.2 ckpt collapsed to 280-300 token repetitive boilerplate
  at greedy decode without it. 1.1 breaks the loops; 1.0 reproduces
  the legacy behavior for parity testing.

### Multi-scheme recall retrieval audit (`scripts/audit/recall_retrieval_audit.py`)

Standalone evaluator for measuring recall query quality across
multiple retrieval schemes. Takes a recall-sample JSONL (gold
queries from pass3c output, OR model-generated predictions) and
outputs aggregate Hit@k / MRR / IoU per scheme.

Schemes:
- `bm25_keyword` â€” text BM25, no time filter (pure lexical baseline)
- `bm25_time` â€” text BM25 + `query.time_range` filter (current v11
  default)
- `dense_text` â€” sentence-transformer (default `all-MiniLM-L6-v2`)
  cosine over think texts
- `hybrid` â€” BM25 + visual cosine; visual encoder is configurable
  (`--visual_encoder siglip|qwen3vl`)

Disk cache (`--cache_dir`) keys text embeddings on
`(video_id, chunk_idx, sha1(text)[:8])`. Visual cache keyed on
`(video_id, chunk_idx)` â€” frames are deterministic at
`AGENT_CHUNK_SEC` granularity, and ViT is frozen
(`tune_mm_vision=False` in `argument.py:19`) so embeddings are
stable across the entire SFT run family. Single-GPU is sufficient:
first cache build takes ~5 min on A100 for 50K embeddings;
subsequent runs are seconds.

### Robust JSON parsing (`scripts/audit/recall_retrieval_audit.py:_extract_query_json`)

Aligned to the canonical pattern from
`pass1a_evidence.parse_evidence_result` (commit 8c71b6e):
- Capture full `<query>...</query>` body (greedy + DOTALL). Earlier
  non-greedy `\{.*?\}` regex stopped at the first `}` for nested
  JSON like `{"query": "...", "time_range": {...}}` and dropped the
  outer object.
- Strip leading ` ```json ` / ` ``` ` and trailing ` ``` ` markdown
  wrappers before `json.loads` â€” the model occasionally fences
  output even inside the `<query>` tag.
- Brace-balance fallback (find first `{`, match to depth 0) so
  partially-malformed JSON still parses where possible.

## Open items (deferred)

The eval-debug audit identified additional issues that aren't
addressed in this revision. Tracked for the next milestone:

1. **Pass2 non-think conversion** â€” same logic as the pass3
   conversion, applied to the per-chunk think generation
   (`pass2_rollout.py:194` and the compress summary call at line
   502). Highest-leverage remaining speedup.
2. **Hard negatives for compress trigger** â€” pair each compress
   sample with a "no trigger" silent sample from the same memory
   state. Forces the model to actually condition on the trigger
   token rather than just learning the keyword distribution under
   memory pressure.
3. **Auxiliary action classification head** â€” at the position
   right after `<think>`, run a binary classifier predicting silent
   vs respond/compress/recall. CE loss directly supervises the
   decision boundary, decoupled from text generation.
   (MMDuet/ViSpeak/Dispider pattern; 4/8 streaming-VLM papers in
   the surveyed set use this.)
4. **GRPO/GDPO compress reward audit** â€” verify
   `trainer/grpo.py` and `gdpo_advantage.py` reward shaping
   includes compress action emission. If reward doesn't reward
   compress emit, the SFT-fixes-format-then-RL-tunes-timing plan
   doesn't work.
5. **Train/eval message format identity check** â€” write a diff
   script that takes one pass3c sample, renders messages via the
   SFT loader and via `streaming_vllm`, and diffs them. Catches
   silent OOD bugs from formatting drift.
6. **Phase 3 RTVU streaming eval entry** â€” copy the OVO entry,
   adapt dataset class.
7. **Recall in eval** â€” currently `allow_recall=False`. After SFT
   learns recall, wire `streaming_vllm` to a `Retriever` instance
   and enable.
