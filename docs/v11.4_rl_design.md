# ThinkStream v11.4 RL Design + Changelog

This document covers two things:

1. **The mathematical principles** behind ThinkStream's RL stage
   (reward decomposition, GRPO vs GDPO advantage aggregation,
   group-normalization choice, mask semantics).
2. **The v11.4 increment** over v11.3 — three reward bug fixes and the
   new selectable GRPO advantage mode.

The implementation is in `thinkstream/trainer/grpo.py` (rollout +
reward + loss orchestration) and `thinkstream/trainer/gdpo_advantage.py`
(pure-tensor advantage aggregation; standalone-importable for
unit testing).

---

## 1. RL setup and notation

A *rollout* is a single trajectory of length $T$ chunks generated by
the policy $\pi_\theta$ from one prompt. For prompt $p$, we sample a
**group** of $G$ independent rollouts $\{\tau_p^{(1)}, ..., \tau_p^{(G)}\}$
and use within-group differences as advantage signal — the GRPO/GDPO
family of algorithms (no value model, no critic, no per-timestep
discounting).

**Notation:**

| Symbol | Meaning |
|---|---|
| $N$ | number of distinct prompts in a training batch |
| $G$ | group size (rollouts per prompt) |
| $B = N \cdot G$ | total rollouts (= shape[0] of all reward tensors) |
| $K$ | number of reward components ($K = 8$ in v11.4) |
| $r_k(i)$ | raw reward for component $k$ on rollout $i \in [0, B)$ |
| $m_k(i) \in \{0,1\}$ | applicability mask: 1 if component $k$ applies to rollout $i$ |
| $w_k$ | scalar weight for component $k$ ($\sum_k w_k$ ≠ 1 by design) |
| $g(i) \in [0, N)$ | group index of rollout $i$ |
| $A(i)$ | scalar advantage for rollout $i$, used in policy gradient |

The standard policy-gradient loss with advantage:

$$
\mathcal{L}_{\text{policy}}(\theta) = -\mathbb{E}_{i \sim \text{batch}}\bigl[A(i) \cdot \log \pi_\theta(\tau_i \mid p_{g(i)})\bigr]
$$

There's also a KL constraint to the SFT reference policy
$\pi_{\text{ref}}$ (the v11.3 ckpt) to prevent reward hacking, but
that's standard PPO/GRPO and orthogonal to the design questions
addressed here.

## 2. Reward components in v11.4

Eight components live in `gdpo_advantage.REWARD_DICT_KEYS`. Each is
computed by a function in `grpo.py` and packed into a `[B]` tensor
plus a `[B]` mask. The columns are stacked into `rewards_per_func`
(dict) and `rewards_masks: [B, K]`.

| $k$ | Component | Sign | Mask = 1 when | Source of truth |
|---|---|---|---|---|
| 1 | `format` | $[0, 1]$ | always | regex on chunk output |
| 2 | `correctness` | $[-0.2, 1]$ | `gold_answer` present | `metadata.gold_answer` |
| 3 | `timing` | $[0, 1]$ | rollout responded **and** $\text{gt\_chunk}$ known | `metadata.support_chunks → gt_chunk_idx` |
| 4 | `silent_quality` | $[-0.6, 0.5]$ | always | rule (silent vs response) |
| 5 | `recall_quality` | $[-0.3, 1]$ | rollout fired recall | rule (JSON well-formed + no answer leak) |
| 6 | `recall_hit_rate` | $[-0.2, 1]$ | recall fired **and** support known | $\frac{\lvert\bigcup \text{returned} \cap \text{support}\rvert}{\lvert\text{support}\rvert}$ |
| 7 | `range_tightness` | $[-0.2, 1]$ | recall used `time_range` **and** support known | $(1 - \frac{\text{width}}{T_{\text{vid}}}) \cdot \text{coverage}$ |
| 8 | `overflow_pen` | $\{-1, 0\}$ | always | `memory_token_count > budget` |

The default weights $w_k$ (post group-norm, so they control *relative
pull*, not magnitude) live in
`gdpo_advantage.DEFAULT_REWARD_WEIGHTS`:

```
correctness     0.30        (primary outcome)
silent_quality  0.18
timing          0.18
format          0.09
overflow_pen    0.10        (sparse Memex-style penalty)
recall_quality  0.05        (query JSON syntax)
recall_hit_rate 0.07
range_tightness 0.03
```

The weighted scalar reward per rollout (used by GRPO mode and as a
logging signal even in GDPO mode):

$$
R(i) = \sum_{k=1}^{K} w_k \cdot r_k(i) \cdot m_k(i)
$$

## 3. Two advantage-aggregation modes

v11.4 makes the aggregation *mode* a selectable hyperparameter via
`advantage_mode ∈ {"gdpo", "grpo"}` (CLI flag passed to the
`advantages` slyme node).

### 3.1 GRPO (DeepSeekMath formulation, baseline)

Aggregate components first → group z-norm second.

For each rollout $i$ in group $g$:

$$
R(i) = \sum_{k} w_k \, r_k(i) \, m_k(i)
$$

Group statistics:

$$
\mu_g = \frac{1}{G} \sum_{j \in g} R(j)
\qquad
\sigma_g = \sqrt{\frac{1}{G} \sum_{j \in g} (R(j) - \mu_g)^2}
$$

Advantage:

$$
A_{\text{GRPO}}(i) = \frac{R(i) - \mu_{g(i)}}{\sigma_{g(i)} + \epsilon}
$$

with $\epsilon = 10^{-4}$. After this, $\sum_{i \in g} A(i) \approx 0$
exactly (within each group), and the standard deviation across the
full batch is approximately 1.

**Properties:**
- One $z$-score per rollout. Simple, well-understood, the standard
  baseline most papers compare against.
- Implicitly assumes intra-group reward variance is well-conditioned —
  true when one outcome reward (correctness) dominates.
- Lossy when components have very different variance scales: a sparse
  $\pm 0.2$ component contributes negligibly to $R(i)$'s spread, so
  its contribution to the gradient signal is mostly washed out by
  the dominant signal.

### 3.2 GDPO (NVIDIA arxiv 2601.05242, default)

Per-component group-norm first → weighted sum → batch-whiten last.

**Step 1 — per-component group-norm (mean-only):**

For component $k$ and rollout $i$ in group $g$:

$$
\mu_g^{(k)} = \frac{\sum_{j \in g} r_k(j) \, m_k(j)}{\max(1, \sum_{j \in g} m_k(j))}
$$

$$
\widetilde{a}_k(i) = m_k(i) \cdot \bigl(r_k(i) - \mu_{g(i)}^{(k)}\bigr)
$$

(Implementation: `nanmean` over the group with `mask=0` rows
NaN-filled, then the mean replaces NaN-only groups with 0.)

**Step 2 — weighted sum:**

$$
\widetilde{R}(i) = \sum_{k} w_k \cdot \widetilde{a}_k(i)
$$

**Step 3 — batch-whiten:**

$$
A_{\text{GDPO}}(i) = \frac{\widetilde{R}(i) - \mu_B}{\sigma_B + \epsilon}
$$

where $\mu_B$, $\sigma_B$ are computed over the full batch (not just one
group).

**Why this design:**

- *Per-component group-norm* gives each reward channel an independent
  pull. A sparse $\pm 0.2$ component will produce $\widetilde{a}_k(i)
  \in [-0.2, 0.2]$ within a group regardless of the magnitude of the
  dense components. After multiplying by $w_k$ it pulls advantage by
  $w_k \cdot 0.2$ — an amount you can directly set by tuning $w_k$.
- *Mean-only normalization* (no division by $\sigma_g^{(k)}$): for
  bimodal reward distributions (e.g., format $\in \{0, 1\}$ depending
  on whether all chunks parse) the empirical $\sigma$ is unstable
  across groups; dividing by it injects high-variance noise into
  advantages that doesn't carry information about which rollout was
  better. ReMemR1 (arxiv 2509.23040) made the same call
  (`grpo_use_adv=False`).
- *Batch-whiten at the end* keeps the advantage scale stable across
  runs with different reward weights or new components added — the
  optimizer's effective learning rate doesn't drift when you tune $w_k$.

### 3.3 Choosing between modes

| Use **GDPO** when | Use **GRPO** when |
|---|---|
| Sparse signals are meaningful but bimodal (the v11 design assumption) | One outcome reward dominates (correctness > 0.7 weight) |
| You want each component to pull policy independently | You want the standard DeepSeekMath baseline for a paper / ablation |
| Mask rates per component are very different | Components have similar variance scales |

The v11.4 default is `gdpo` because the recall sub-rewards
(quality / hit_rate / range_tightness) have mask rates around 30-50%
while correctness is always-on; per-component group-norm prevents the
correctness gradient from dominating just because it has more rows
contributing variance.

## 4. Mask semantics — v11.4 fix

The mask $m_k(i)$ encodes "is component $k$ applicable to rollout $i$?"
The v11.3 implementation conflated two semantically different cases
into mask=0:

- $C_1$: the component genuinely doesn't apply (e.g., `gold_answer`
  unknown so we can't score correctness, or the rollout never fired
  recall so there's no query to evaluate).
- $C_2$: the component fired but failed in the worst possible way
  (e.g., recall fired but the retriever returned nothing, or the
  recall query's `time_range` didn't cover any gold chunk).

In v11.3, both gave mask=0 → component dropped from the per-component
group mean → the policy got no gradient signal for $C_2$ failures.
**This is exactly the failure mode the policy must learn to avoid.**

v11.4 separates them:

| Case | v11.3 | v11.4 |
|---|---|---|
| Component doesn't apply ($C_1$) | mask=0, $r=0$ | mask=0, $r=0$ (unchanged) |
| Recall fired but `returned_chunks=∅` | mask=0 | mask=1, $r=-0.2$ |
| `time_range` covers 0 gold chunks | mask=0 | mask=1, $r=-0.2$ |
| First-response timing | always uses $\text{first}$ | uses $\text{last}$ (refinement-tolerant) |

The negative reward at the failure case becomes a real signal — it's
below the group mean, so `aggregate_gdpo` produces a negative
advantage for that rollout; the policy gradient pushes the log-prob
of *that specific query/range output* down.

## 5. Reward bug fixes in v11.4

### 5.1 Timing reward used first-response

`_compute_time_reward` measured the distance between the
**first** chunk that produced any parseable response and
`gt_chunk_idx`. Streaming agents that respond then refine in later
chunks (perfectly natural and arguably better behavior) were
penalized — the first response was usually less polished than the
final commitment.

**Fix:** `_scan_responses_for_answer` now also returns
`last_response_chunk_idx`. `calc_rewards` passes the *last* chunk's
index to `_compute_time_reward`. `first_answer` still gates
correctness so the model can't game by spamming until something
matches.

### 5.2 `recall_hit_rate` masked failed recalls

If the model emitted `<action>recall</action>` but the retriever
returned nothing (e.g., the query had zero overlap with archive
text), `_compute_recall_hit_rate` returned `None`. Caller treated
None as "no signal" → mask=0 → the rollout got no gradient signal
for an obviously-bad recall query. The model couldn't learn to
write better queries because its worst queries were invisible to
the loss.

**Fix:** new `recall_fired` keyword arg. When fired AND
`returned_chunks=∅`, return `-0.2` (an explicit penalty). The
caller still passes `did_recall` to mask=1 in that case, so the
penalty reaches advantage.

### 5.3 `range_tightness` masked zero-coverage queries

Same pattern: `_compute_range_tightness_reward` returned `None`
when the query's `time_range` covered no gold chunks. That's the
worst possible time-range output (asking for a window where the
answer doesn't live), and v11.3 hid it.

**Fix:** return `-0.2` when coverage equals zero. mask logic
unchanged (it was already `did_recall and range_tight is not None`,
and `-0.2 is not None` is True).

## 6. Implementation map

```
thinkstream/trainer/grpo.py:
  rollout()                        — chunk-lockstep rollout (HF or vLLM)
  _scan_responses_for_answer()     — extract first/last response per gen [v11.4]
  _compute_response_reward()       — correctness
  _compute_recall_hit_rate()       — hit-rate, recall_fired arg [v11.4]
  _compute_recall_quality_reward() — query syntax + leak check
  _compute_range_tightness_reward()— range tightness, neg on coverage=0 [v11.4]
  _compute_format_reward()         — chunk format compliance
  _compute_time_reward()           — timing (now uses last response) [v11.4]
  _compute_silent_quality_reward() — silent vs response correctness
  _compute_overflow_penalty()      — Memex-style memory overflow signal
  calc_rewards()                   — assembles all 8 components + masks
  advantages()                     — slyme node, dispatches to aggregate_advantages [v11.4]

thinkstream/trainer/gdpo_advantage.py:
  REWARD_DICT_KEYS                 — (8,) column ordering
  DEFAULT_REWARD_WEIGHTS           — w_k dict
  per_reward_group_norm()          — Step 1 of GDPO (mean-only group-norm)
  aggregate_gdpo()                 — full GDPO pipeline
  aggregate_grpo()                 — DeepSeekMath baseline [v11.4]
  aggregate_advantages()           — dispatcher [v11.4]
```

**Sanity test** (run on CPU, no model needed):
```bash
python3 -c "
from thinkstream.trainer.gdpo_advantage import (
    aggregate_gdpo, aggregate_grpo, aggregate_advantages, REWARD_DICT_KEYS,
)
import torch
B, G = 8, 2
rewards = {k: torch.randn(B) for k in REWARD_DICT_KEYS}
masks = torch.ones(B, len(REWARD_DICT_KEYS))
adv_g, _ = aggregate_gdpo(rewards, masks, G)
adv_p, _ = aggregate_grpo(rewards, masks, G)
# GRPO group z-norm: each group sums to 0 within tolerance
for n in range(B // G):
    chunk = adv_p[n*G:(n+1)*G]
    assert abs(chunk.sum().item()) < 1e-3
print('GDPO adv std =', adv_g.std().item())  # ≈ 1.0 (batch-whitened)
print('GRPO adv std =', adv_p.std().item())  # ≈ 1.0+ (sum of group z-scores)
"
```

## 7. Design philosophy — what RL is actually doing here

The ThinkStream RL stage is **not** trying to teach the model new
visual/linguistic capabilities (SFT did that). It's trying to teach
the model **decision timing** in the streaming agent setting:

- *When* to stay silent vs respond (silent_quality + correctness +
  timing).
- *When* to recall (recall_quality + hit_rate + range_tightness).
- *When* to compress (overflow_pen).

These are decision-policy questions, not capability questions.
That's why the reward design emphasizes:

1. **Outcome correctness as the dominant pull** ($w = 0.30$). Most of
   the policy update comes from "did the eventual answer match
   gold." This is the only reward whose signal generalizes cleanly to
   unseen test sets.

2. **Sparse process penalties for explicit failure modes** (recall
   hit_rate $-0.2$, range_tightness $-0.2$, overflow_pen $-1$).
   These are specifically the cases the model wouldn't otherwise
   learn from *outcome alone* because they're dominated by random
   correctness fluctuation. The penalties make the failure mode
   directly visible to the policy gradient.

3. **No process reward on `<think>` content quality.** SFT span weights
   already supervise think content. Adding an RL reward there would
   double-supervise and risk the SFT and RL objectives diverging
   (SFT teaches "be specific and visual"; an RL think-quality reward
   based on, e.g., length would push the opposite). $w_{\text{think}} = 0$
   is intentional.

4. **Format reward kept low ($w = 0.09$).** SFT already teaches format
   via the chat template loss mask; the RL format reward is mostly a
   safety net against a degenerate solution where the model emits
   well-scored *content* but malformed action tags.

5. **The recall signals depend on `support_chunks`, which is a
   teacher artifact.** This is a known generalization risk:
   `support_chunks` is generated by 397B teacher in pass3a, so the
   RL policy's retrieval behavior is upper-bounded by what the
   teacher could correctly identify as gold evidence. ReMemR1 /
   MemAgent / DeepEyes all use outcome-only rewards for retrieval to
   avoid this. v11.4 keeps the support_chunks-based rewards but
   weights them low ($w_{\text{recall\_quality}} + w_{\text{hit\_rate}} +
   w_{\text{range\_tightness}} = 0.15$) and surfaces the failure
   penalty so the dominant signal is still correctness — the
   support_chunks rewards are scaffolding, not the main objective.
   v11.5 (deferred) considers moving them to eval-only metrics.

## 8. Open / deferred to v11.5

These came up in the audit and reward-redesign discussion but are
**not** addressed in v11.4:

1. **Move support_chunks rewards to eval-only**: drop
   `recall_hit_rate` + `range_tightness` from training, leaving only
   correctness as the retrieval signal. Aligns with ReMemR1 /
   MemAgent / DeepEyes; biggest generalization win.

2. **Drop `silent_quality`'s HLD-binary branch**: the
   `answer_form == "binary"` special case encodes OVO HLD task
   structure. Won't transfer cleanly to non-binary OVO families.

3. **Continuous overflow penalty**: replace binary
   $r_{\text{overflow}} \in \{-1, 0\}$ with
   $r_{\text{overflow}} = -\max(0, \text{mem}/\text{budget} - 0.7)$
   so the policy learns memory efficiency, not just "stay below
   budget."

4. **Format reward ablation**: 1000-step train with $w_{\text{format}}$
   set to 0 vs 0.09. If correlation with SFT format loss > 0.8,
   remove the RL format reward as redundant.

5. **Auxiliary action classification head** (MMDuet / ViSpeak /
   Dispider pattern). Decouples action-keyword decision boundary
   from the main LM token loss. Architectural change, deferred.
