# v12.0 — Migration to Official Qwen Tool Protocol + Agentic Reframing

> **状态**: 设计稿，未实施。
> **前置阅读**: `docs/v11.4_rl_design.md`、`docs/v11.3_sft_run_postmortem.md`、`memory/project_v113_changes.md`。
> **决策依据**: 3 轮源码级调研——`/tmp/agent-sft-research/` 中 19 个仓库。

## 1. 动机：v11.x 不稳的根本原因

v11.3/v11.4/v11.5 三轮 SFT 在同一个失败模式上反复打转：

| 现象 | step 100→1240 |
|------|---------------|
| `train/action_argmax_acc_compress` | 0 → **1.0** ✅ |
| `eval/action_acc_compress` | 0 → **0** ❌ |
| `eval/silent_acc` | 1.0 → **0.86**（过拟合后还在跌） |
| 试过的修复 | sample_weight rebalance / class-balanced sampler / focal+α / smart_init / span_weight=8 |

**所有修复都治标不治本**。源码调研结论（`/tmp/agent-sft-research/` 19 仓库）：

| 维度 | ThinkStream v11.x | 业内（DeepEyes/ReTool/MMDuet）|
|------|-------------------|-------------------------------|
| 工具协议 | 自造 `<action>X</action>` 词表，5 类 softmax | 官方 `<tool_call>{json}</tool_call>` free-form text |
| Role 数量 | 3 种（system/user/assistant） | **4 种**（+ tool） |
| 工具注册 | 写死在 system prompt 文本 | `<tools>` JSON schema 块 |
| 工具结果回注 | user role | **tool role**（SFT）/ user role（RL env hack） |
| SFT loss | span 权重 5 类 + sample 权重 + focal+α | **vanilla CE**，仅 assistant 整段=1 |
| 稀有动作 | 4 层 rebalance（采样 + sample_w + span_w + focal+α）| **不做**——RL 修偏 |
| 动作选择 | softmax over {silent, response, recall, compress} | answer (terminal) + tool (recall) + system event (compress) **3 层架构分离** |

**根本问题**：把 compress（资源管理）和 silent/response/recall（语义决策）放进同一个 softmax 头，让模型学一个**它无法观测的变量**（context 占用率）。这是不可学习目标，无论怎么调权重都治不了。

**v12.0 决策**：迁移到官方 Qwen2.5 tool protocol，重新建模为 agentic VLM。

## 2. 架构总览

### 2.1 三层职责划分

| 层 | 内容 | 触发方 |
|------|------|--------|
| **Answer**（终止）| `<answer>response_text</answer>` 或空（silent） | LM 自主决定 |
| **Tool**（工具调用）| `<tool_call>{"name":"recall",...}</tool_call>` | LM 自主决定 |
| **System Event**（环境事件）| `<compress_trigger range="..."/>` 注入 user prompt | 环境（chunk_count_since_last_compress ≥ K） |

**关键变化**：
- silent / response → 同一个 `<answer>` 标签的不同 payload（空 vs 文本）
- recall → 工具调用，不是动作
- compress → 系统事件 + 模型生成 summary，不是动作

### 2.2 Action 词表彻底消失

| v11.x | v12.0 |
|-------|-------|
| 5 类 softmax `{silent, response, recall, compress, recall_response, ...}` | **无动作词表** |
| `<action>compress</action>` 单独训 keyword | 无 keyword 概念 |
| `ACTION_WEIGHTS` / `SPAN_WEIGHTS["action"]=8` | 全部移除 |

模型生成的是自然 free-form text，工具/答案/思考都是 BPE-friendly 标签。

## 3. 单步 vs 多步建模决策

### 3.1 SFT：保持单步

**理由（从 `/tmp/agent-sft-research/DeepEyesV2/.../multiturn_sft_dataset.py:170` 验证）**：

token-level CE 是位置可加的：
```
loss(N decisions in 1 trajectory) == loss(N samples each 1 decision)
```

只要 loss mask 正确（assistant=1, others=0），单步打包等价于多步。SFT 单步的 **唯一损失** 是 KV cache 重用（rollout 时无法跨样本复用）——但 SFT 不做 rollout，所以损失为零。

**收益**：
- 数据 IO 简单（每条样本是独立 (memory, video_window) → 1 assistant turn）
- batch 内可以混合不同视频的不同时刻，class-balanced sampler 直接生效
- 不需要 trajectory packing 的复杂逻辑

### 3.2 RL：必须多步

**理由**：单步 GRPO 无法做跨 turn credit assignment。例：第 5 chunk 调 recall 是为第 8 chunk 答对铺垫——单步 RL 在第 5 chunk 看到 reward 时，第 8 chunk 还没发生，无法把成功反传给 recall 决策。

**多步 trajectory rollout**：
- 一条 episode = 该视频从 t=0 到 t=T 的所有 chunks
- 每个 chunk 模型生成 assistant turn（think + tool_call/answer）
- tool 结果通过 user role 注入下一 turn
- final reward 在 episode 结束时计算，GRPO 用 group advantage 反传所有 turn

**实现路径**：fork DeepEyesV2 verl recipe（`/tmp/agent-sft-research/DeepEyesV2/reinforcement_learning/recipe/`），不要重新发明轮子。

### 3.3 流视频特殊性：滑动窗口与多模态注入

DeepEyes 单图任务每 turn 视觉内容固定，但流视频每 chunk 视觉内容变化。这要求 multi-turn rollout 时：

| 时刻 | user message 内容 |
|------|------------------|
| t=0（初始）| `<visual_window>frames[0:12]</visual_window><user_input>query</user_input>` |
| t=k（系统事件）| `<visual_window>frames[k:k+12]</visual_window>`（滑动）+ 可选 `<recall_result>` 或 `<compress_trigger>` |

每个 turn 的 user role 内容**完全重写**（视觉窗口滑动 + memory 状态更新），不是单纯追加。这与 DeepEyes 多图工具调用本质相同（每次 zoom_in 也是新视觉内容），所以官方 chat_template 直接适用。

## 4. SFT Loss 设计

### 4.1 计算方式

```python
# 来自 /tmp/agent-sft-research/DeepEyesV2/.../multiturn_sft_dataset.py:159-176
if role == "assistant":
    loss_mask = [0] * gen_prompt_len + [1] * (msg_len - gen_prompt_len)
else:
    loss_mask = [0] * msg_len  # system / user / tool 全部 mask
```

**ThinkStream v12.0 完全照搬**：

```python
loss = CrossEntropyLoss(reduction="none")(logits, labels)
loss = (loss * loss_mask).sum() / loss_mask.sum()
```

|---------|------|

| `ACTION_WEIGHTS = {silent:1.2, compress:1.8, ...}` | **删除** |
| `_get_sample_weight()` 复杂 sequence_type 分支 | **删除** |
| `focal_alpha_action` / `focal_gamma` / `alpha_softening` | **删除** |
| `smart_init_special_token_embeddings()` | **删除**（不再有新 special token）|
| `_build_token_loss_weight()` | **删除** |
| `_extract_eval_positions()` action_keyword 逻辑 | **删除** |

### 4.3 保留清单

| 组件 | 理由 |
|------|------|
| `ClassBalancedDistributedSampler` | 视频/sample_type 分布平衡，不是 token 级权重，安全 |
| `class_balance_smoothing=0.85` | 同上 |
| 视觉 token 不在 loss | 业内一致（user role 自动 mask） |
| Per-sample audit log | 调试用 |

### 4.4 为什么不再需要 rebalance

**业内证据（来自调研）**：
- DeepEyesV2 cold-start: `qwen2_5vl_full_sft_final_100k.yaml` 默认 LLaMA-Factory，**无任何 class weight**
- ReTool: `verl.trainer.fsdp_sft_trainer` 默认配置，**无 reweight**
- MMDuet2 §4.3.1 明确写："an intuitive solution is to encourage the model to generate more favorable interactions through GRPO training"——**SFT 不修偏**

我们 v11.x 的 4 层 rebalance 是 industry convention 的 4 倍冗余。回归 vanilla CE 后预期：
- train/eval gap 缩小
- 过拟合速度变慢
- compress 和 recall 的 free-gen trigger 率上升（被过度采样压制的稀有动作释放）

## 5. RL Reward 设计

### 5.1 当前 v11.4 reward（来自 `project_v114_rl` 记忆）

```
reward = format(0.09) + acc + range_coverage + recall_failure(-0.2)
       + range_coverage=0(-0.2) + 各类细分 timing/last_response 等
```

8+ 组件，gradient 信号互相 cancel。

### 5.2 v12.0 reward（DeepEyesV2 风格）

```python
def compute_reward(trajectory):
    # 1. Outcome (主信号)
    acc = teacher_judge(trajectory.final_answer)  # ∈ [0, 1]

    # 2. Format (基础闸门)
    fmt = -1.0 if any_tag_unbalanced(trajectory) else 0.0

    # 3. Anti-spam (DeepEyesV2 search variant 模式)
    n_recall = trajectory.count_tool_call("recall")
    spam_factor = 1.0 - 0.1 * (n_recall > 0)  # 单次调用就扣 10%，惩罚多调

    # 4. Compress turn 单独评 summary 质量
    summary_q = 0.0
    for turn in trajectory.compress_turns:
        summary_q += rouge_or_coverage(turn.summary, turn.gold_summary)
    summary_q /= max(1, len(trajectory.compress_turns))

    # 总 reward (3 组件 + 1 conditional)
    return 0.8 * acc * spam_factor + 0.2 * fmt + 0.2 * summary_q
```

### 5.3 删除清单

| 当前 reward 组件 | v12.0 处理 |
|----------------|-----------|
| `recall_failure -0.2` | **删除**——recall 失败 → 答案错 → acc=0 隐式吸收 |
| `range_coverage=0 -0.2` | **删除**——同上 |
| `timing_use_last_response` | **删除**——多步 RL 自然处理 |
| `aggregate_grpo` ablation | **保留为 ablation flag**，默认关 |

### 5.4 为什么不要 conditional tool bonus

DeepEyes 的演化：
- v1: `1.2 * tool_reward if (used_tool AND correct)` — 显式奖励工具使用
- v2: 删除 tool_reward，改成 `(1 - search_penalty) * acc` — 反向**惩罚**工具滥用

v2 的反思：tool bonus 在 SFT 已经会用工具的情况下**会引发 spam**。我们的 SFT 应该已经会用 recall（只要 free-gen gate 验证通过），不需要 v1 的 bonus 模式。

### 5.5 新增信号：summary quality 是 compress 学习的唯一来源

因为 compress 是系统触发，**没有 timing 决策可以 reward**。对 compress turn 的优化全靠 summary 质量：
- ROUGE-L 与 gold summary 比较
- 或 coverage：summary 是否包含 gold range 内的关键事件
- 权重 0.2，与 format 平级

## 8. Eval 重构

### 8.1 当前 eval 指标（来自 trainer.py）

```
eval/action_accuracy
eval/action_acc_{compress, silent, response, recall_query, recall_response}
eval/silent_eos_rate
eval/format_compliance(_<class>)
eval/post_action_acc_silent
eval/summary_argmax_acc
eval/query_argmax_acc
eval/response_argmax_acc
```

8 类指标，全部基于 teacher-forced argmax。

### 8.2 v12.0 eval 指标

**核心变化**：去掉 action_acc 系列（不再有 action 词表），改为 free-gen rollout 指标。

| 新指标 | 定义 | 替代 |
|--------|------|------|
| `eval/free_gen_tool_rate_recall` | 100 样本中 LM 自由生成调 recall 的比例 | `action_acc_recall_query` |
| `eval/free_gen_summary_emitted_rate` | compress_trigger 注入后 LM 真正生成 summary 的比例 | `action_acc_compress` |
| `eval/free_gen_answer_rate` | LM 生成非空 answer 的比例 | `action_acc_response` |
| `eval/free_gen_silent_rate` | LM 生成空 answer 的比例 | `action_acc_silent` |
| `eval/answer_correctness` | 教师评判答案正确率 | （新增，主指标）|
| `eval/format_compliance` | tag 平衡 + JSON parse OK | 保留 |
| `eval/recall_range_iou` | recall 范围与 gold 的 IoU（diagnostic only）| （新增）|
| `eval/summary_rouge_l` | summary 与 gold 的 ROUGE-L | （新增）|

### 8.3 Eval 实现

**Teacher-forced eval（loss 计算）**：保留，但只用作 `eval_loss` 一个数值，不再拆 per-class action_acc。

**Free-gen eval**：复用 `eval_baseline_vllm.py`，扩展输出统计：
```python
# 新增统计
n_recall_emitted = count_tool_calls(rollout, "recall")
n_compress_summary = count_after_trigger(rollout, "<summary>")
n_answer_nonempty = count_nonempty_answer(rollout)
```

### 8.4 SFT→RL go/no-go gate

**Gate A**: `free_gen_tool_rate_recall ≥ 0.5 × training_freq`（DeepEyes v1 line 261 same logic）
**Gate B**: `free_gen_summary_emitted_rate ≥ 0.95`（compress 是确定性触发，几乎必生成）
**Gate C**: `format_compliance ≥ 0.9`
**Gate D**: `eval_loss < threshold`（监控用，不 gate）

任一 gate 不通过 → 回 SFT 排查；全通过 → 进 RL。
